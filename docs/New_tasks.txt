# Phase 4 Tasks — Edge Gateway + Cloudflare Integration (Core-ID Hostnames, V1 Single-Owner Model)
# Start at Task 680

## Scope

This phase introduces the **Core Edge Gateway** and integrates **Cloudflare Tunnel** as the public ingress layer, using a **stable per-Core public hostname model**.

### Core Hostname Model

Each Core instance must have a stable unique ID and public hostnames derived from it.

Canonical pattern:

```text
<core-id>.hexe-ai.com
api.<core-id>.hexe-ai.com
```

Example:

```text
7f3c91ab42de8f10.hexe-ai.com
api.7f3c91ab42de8f10.hexe-ai.com
```

### Goal

- every Core gets a stable unique public identity
- Cloudflare routes traffic to the correct Core by Core-specific hostname
- Core becomes the public entry point for UI and API
- Core routes traffic internally to:
  - Core services
  - Supervisor-managed runtimes
  - Nodes

---

## V1 Ownership Constraint

This phase is intentionally **single-owner / platform-managed only**.

### V1 assumptions
- one Cloudflare owner
- one managed domain base: `hexe-ai.com`
- one Cloudflare token source
- one Cloudflare account context
- one Cloudflare zone context
- Core creates/manages public hostnames only under the platform-owned domain

### V1 non-goals
- no per-user Cloudflare ownership
- no bring-your-own-domain
- no per-tenant token selection
- no multi-tenant zone management
- no user-facing Cloudflare account linking

Future multi-user / multi-tenant support may be added later, but this phase must stay intentionally simple.

---

## Design Rules

- `core_id` must be stable once created
- `core_id` must be URL-safe
- recommended format: **16-character lowercase hex**
- hostname must be derived from persisted Core identity, not regenerated on reboot
- hostname is an address, not the trust mechanism
- trust/auth must still rely on persisted identity, tokens, and stored credentials
